# ApexPowers Supply-Chain, Trust, And Hook Security

This document is the release checklist for community distribution. It keeps supply-chain provenance, hook trust, generated-file ownership, telemetry, and guard false-positive policy in one reviewable place.

## Scope

- This project distributes skills, command prompts, docs, tests, benchmarks, and installer/runtime scripts.
- Plugin manifests stay thin. The plugin manifests must not declare hooks or silently install lifecycle commands.
- Lifecycle hooks are installed only through `apex-init-project-hooks`, and the host still owns the final review / trust decision.
- Future plugin-bundled hooks must remain opt-in and must still surface a trust UI or an equivalent explicit review step before execution.

## Trust-Critical SHA256 Manifest

`docs/supply-chain-manifest.sha256` records byte-level `sha256` values for trust-critical distribution files:

- `.codex-plugin/plugin.json`
- `.claude-plugin/plugin.json`
- `commands/*.toml`
- hook installer and runtime source files
- this document and `NOTICE.md`

The manifest is not an authenticity system by itself. It is a drift detector for release review: when a trust-critical artifact changes, the reviewer must inspect the diff first, then regenerate the manifest.

Installer ownership manifests use normalized text hashes in `files[].hash`. Those hashes are also SHA-256 values and are used by update / uninstall to tell Apex-managed files from user-modified files.

## Generated File Marker And Provenance

Generated hook files use this marker:

```text
Generated by ApexPowers apex-init-project-hooks
```

Provenance is recorded in three places:

- Generated runtime or TOML blocks include the marker, including `# >>> apex-managed-hooks-begin (Generated by ApexPowers apex-init-project-hooks) >>>`.
- Ownership manifests use `managed_by: Generated by ApexPowers apex-init-project-hooks`.
- Manifest entries include `root`, `relative_path`, `kind`, `hash`, and `protected`.

Files without this marker and without a matching manifest hash are treated as user-owned unless `--force` is explicitly passed.

## Hook Command Threat Model

Every hook command is local and deterministic. Hooks do not call external APIs, do not start agents recursively, and do not read credentials for telemetry.

| Hook event / route | Matcher | Assets at risk | Threat | Mitigation |
| --- | --- | --- | --- | --- |
| `SessionStart/default` | default | Local path names, workflow state, review state | Stale or overbroad context could nudge an agent toward unrelated work. | Read-only context builder; no writes; no network; state is scoped to the target project. |
| `UserPromptSubmit/default` | default | User prompt intent | Completion hints could be overinterpreted as permission to skip validation. | Advisory only; Stop gate still checks review and validation evidence. |
| `PreToolUse/safety` | `Bash`, `Shell`, `PowerShell`, `Edit`, `Write`, `MultiEdit`, `apply_patch` | Git history, files, secrets, local paths | Destructive commands, secret path access, or unsafe edits could run before review. | Blocks dangerous command patterns and secret-like paths before execution; false-positive coverage includes normal names such as `tokenizer.py`. |
| `PostToolUse/edit` | `Edit`, `Write`, `MultiEdit`, `apply_patch` | Source files and accidental credentials | A write could create oversized generated code or commit secret-looking content. | Runs line-length and secret-content guards after edits; generated-file exemptions are explicit. |
| `PostToolUse/bash` | `Bash`, `Shell`, `PowerShell` | Local command output and changed files | Shell commands may create files outside the agent's immediate awareness. | Post-use checks inspect relevant paths from tool input and fail closed on secret-looking writes. |
| `PostToolUse/always` | default | Runtime stability | A broad always-on hook could become noisy or non-deterministic. | Keep implementation deterministic and local; do not add network, telemetry, or agent recursion. |
| `Stop/default` | default | Review quality and release state | Agent may finish after code changes without review or validation. | Requires review/validation evidence after code diffs and blocks stale ready-review states. |

## Trust Flow

Codex and similar hosts treat unmanaged lifecycle commands as a security boundary. Adding a hook command to a config file is not the same as trusting it.

Current policy:

- `apex-init-project-hooks` can write the correct host config location.
- It does not write Codex trust state, fake `trusted_hash`, or bypass review / trust prompts.
- Plugin manifests remain hook-free so package installation cannot secretly activate lifecycle commands.
- If plugin-bundled hooks are supported later, they must be opt-in and must require explicit trust UI acceptance before execution.

## Update / Uninstall Safety

The installer treats update / uninstall as destructive operations and protects user-owned files:

- Default mode is dry-run; `--write` is required to change disk.
- `--update` is a manifest-aware reinstall.
- `--uninstall` removes only Apex-managed entries unless `--force` is passed.
- Host config cleanup scrubs Apex hook entries while preserving user hooks.
- Runtime deletion requires a matching hash, a generated marker, or explicit `--force`.
- Protected state files such as `tasks/loops/workflow.md` and `tasks/lessons.md` are kept by default.

Required destructive tests:

- `test_uninstall_removes_managed_files_and_preserves_user_hooks`
- `test_uninstall_uses_lf_normalized_hashes`
- `test_uninstall_keeps_manifest_when_managed_file_is_modified`
- `test_uninstall_manifest_survives_project_move`
- `test_agent_scope_removes_generated_legacy_project_hooks`
- `test_agent_scope_strips_legacy_apex_entries_but_keeps_user_hooks`

## Telemetry Policy

Default: no telemetry.

ApexPowers hooks and distribution checks must not send analytics, usage events, source snippets, prompts, file paths, tokens, or error reports to any external service by default.

Future telemetry requires all of the following:

- explicit opt-in configuration,
- documented fields and destination,
- a local dry-run preview of payload shape,
- tests proving disabled-by-default behavior,
- a redaction pass for secrets, local paths, and prompts.

## Secret/Path Guard False-Positive Policy

Secret and path guards are intentionally conservative, but normal developer names must keep working.

Current required tests:

- `test_pre_tool_use_blocks_secret_path`
- `test_pre_tool_use_allows_secret_substrings_in_normal_names`
- `test_post_tool_use_blocks_secret_content`

The phrase secret/path guard false-positive is intentionally present for distribution checks. Add a positive block test and a false-positive allow test whenever the pattern list changes.
