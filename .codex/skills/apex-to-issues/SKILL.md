# 拆分为 Issues

使用垂直切片（tracer bullets）把计划拆分成可被独立领取的 issues。

issue tracker 和 triage label 词汇表应该已经提供给你；如果没有，请运行 `/setup-matt-pocock-skills`。

## 流程

### 1. 收集上下文

基于对话上下文中已有的所有信息开展工作。如果用户以参数形式传入 issue 引用（issue 编号、URL 或路径），请从 issue tracker 获取该 issue，并阅读其完整正文和评论。

### 2. 探索代码库（可选）

如果你还没有探索过代码库，请先探索，以理解代码当前状态。Issue 标题和描述应使用项目的领域词汇表，并尊重你所触及区域内的 ADR。

### 3. 起草垂直切片

把计划拆分成 **tracer bullet** issues。每个 issue 都是一个从端到端贯穿所有集成层的薄垂直切片，而不是某一层的水平切片。

切片可以是 `HITL` 或 `AFK`。HITL 切片需要人类参与，例如架构决策或设计审查。AFK 切片可以在无人类参与的情况下实现并合并。在可能的情况下，优先选择 AFK 而不是 HITL。

<vertical-slice-rules>
- 每个切片都交付一条穿过每一层（schema、API、UI、tests）的窄而完整的路径
- 已完成的切片本身可以演示或验证
- 优先拆成多个薄切片，而不是少数厚切片
</vertical-slice-rules>

### 4. 向用户提问确认

以编号列表形式呈现拟议拆分方案。对每个切片，展示：

- **标题**：简短的描述性名称
- **类型**：HITL / AFK
- **阻塞项**：必须先完成哪些其他切片（如果有）
- **覆盖的用户故事**：该切片处理哪些用户故事（如果源材料中包含用户故事）

询问用户：

- 粒度是否合适？（过粗 / 过细）
- 依赖关系是否正确？
- 是否有任何切片应该合并或进一步拆分？
- HITL 和 AFK 的标记是否正确？

持续迭代，直到用户认可拆分方案。

### 5. 将 issues 发布到 issue tracker

针对每个已认可的切片，在 issue tracker 中发布一个新 issue。使用下面的 issue 正文模板。这些 issues 被视为已准备好交给 AFK agents，因此除非另有指示，否则发布时应带上正确的 triage label。

按依赖顺序发布 issues（阻塞项优先），这样你可以在“阻塞项”字段中引用真实 issue 标识符。

<issue-template>
## 父级

issue tracker 上父级 issue 的引用（如果来源是已有 issue；否则省略本节）。

## 要构建什么

对这个垂直切片的简洁描述。描述端到端行为，而不是逐层实现。

避免写入具体文件路径或代码片段，因为它们很快会过期。例外：如果某个原型产出的片段能比文字更精确地表达决策（状态机、reducer、schema、类型形状），可以在这里内联该片段，并简短说明它来自原型。请裁剪到富含决策信息的部分；这里不是工作演示，只保留重要片段。

- [ ] 标准 2
- [ ] 标准 3

</issue-template>
